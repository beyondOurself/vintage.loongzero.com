## 开发规范
- 默认最小改动
- 优先修改指定文件
- 不改无关文件
- 仅当改动项目业务代码时，须同步更新 `TODO.md`（文档-only、配置-only 等可不更新）
- `TODO.md` 需要记录：已完成事项、进行中事项、下一步事项
- 涉及业务代码的任务未完成且未同步 `TODO.md` 不视为任务完成

## 环境约束（Windows）
- OS: Windows
- Shell: PowerShell
- 命令兼容：优先 PowerShell 语法

## 代码风格
- 代码应高效、规范，结构清晰、易于理解和维护。
- 命名简洁有意义，采用驼峰（camelCase）或短横线（kebab-case）风格，保持一致。
- 缩进统一为 2 空格，不使用制表符（Tab）。
- HTML/CSS/JS 按功能和结构合理分组，避免重复和大段无效注释。
- 注释应简明扼要，仅在必要处解释复杂逻辑。
- 保持文件结构简洁，目录命名清晰，避免多级嵌套。
- 推荐使用 Tailwind CSS 原子类为主，定制样式集中于独立 CSS 文件。
- 代码提交前确保通过本地检查，避免低级语法错误和拼写问题。

## 提交规范
- 提交含业务代码变更前，须确认 `TODO.md` 已同步对应进度

## 跨设备开发流程（强制）
- 在不同设备开始开发前，先读取 `TODO.md` 了解最近业务进度
- 以 `TODO.md` 作为业务任务状态单一事实来源
- 开发业务代码过程中，我明确要求写入时才写入 `TODO.md`
- 结束含业务代码的开发前，检查 `TODO.md` 是否包含当次相关改动记录
